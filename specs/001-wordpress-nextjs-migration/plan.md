# Implementation Plan: WordPress to Next.js Migration (Omri Meron Studio)

**Branch**: `main` (no feature branch created; single-owner repo) | **Date**: 2026-07-11 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `/specs/001-wordpress-nextjs-migration/spec.md`

## Summary

Rebuild the omrimeron.com WordPress photography portfolio as a statically generated Next.js (App Router) + React + TypeScript site. All content (site identity, 12 navigation tabs, page copy, gallery definitions with image slot lists, 28 client names) is generated mechanically from the WordPress database dump by `scripts/extract_content.py` into a typed local content layer whose document shapes mirror future Sanity.io schemas. **The generated `content/` files, the test fixture, and the media manifest already exist in the repo** (the extractor ran and its invariants passed); implementation consumes them, it does not re-create them. Pages render from that layer through a single content-access module, so the later Vercel + Sanity hookup swaps one data source without touching components. No image binaries are migrated; every image slot renders a neutral placeholder that preserves its source attachment reference.

## Technical Context

**Language/Version**: TypeScript 5.x, Node.js 20+ (LTS)

**Primary Dependencies**: Next.js 15 (App Router, static generation), React 19, Embla Carousel (`embla-carousel-react`) for the fullscreen slider

**Storage**: Local structured content files (typed TS objects under `content/`), shaped as Sanity-style documents; no database

**Testing**: Vitest + React Testing Library (content-layer integrity tests, navigation/slider component tests); end-to-end validation scenarios documented in quickstart.md

**Target Platform**: Static web (any Node host; Vercel intended later), modern evergreen browsers, mobile + desktop

**Project Type**: Single web application (frontend only, no backend)

**Performance Goals**: Fully static output (`next build` prerenders every route); Lighthouse performance ≥ 90 on portfolio pages (trivial with placeholder images)

**Constraints**: Zero source image binaries in the repo or build output (spec SC-006); content edits must not require component edits (spec SC-007/FR-007); old WordPress URL paths must resolve (FR-008), including the Hebrew slug `צילום-תעשיה-עמרי-מירון` for Industry

**Scale/Scope**: 13 routes (12 nav pages + Our Clients), 10 slider pages totaling ~174 image slots, 15 standalone gallery definitions (~182 slots) captured as data only, 28+ client entries

│                           # tests/fixtures/source-inventory.json + media-manifest.json.

    ├── SiteHeader.test.tsx

**Structure Decision**: Single Next.js application at the repository root (the repo `omrimeron-studio` exists solely for this site). No backend, no monorepo split: content is local data, so a separate API project would be unjustified complexity. The `lib/content.ts` module is the seam for the future Sanity swap.
